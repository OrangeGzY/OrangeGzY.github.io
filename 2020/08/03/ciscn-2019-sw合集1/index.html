ciscn_2019_sw_5

这题被师傅们的trick惊了，果然还是我太菜了

题目概览

1.glibc2.27 tcachebin。

2.固定申请0x70，读两次，前0x8是title，后0x68是content。每次add完直接自动show

3.最多del三次

4.这题构造的实在是没想到，我愿称之为螺旋链。

思路

只能del三次，所以我们选择劫持tcache结构体，同时造出0x100的fakechunk，free进unsortbin。

1.首先一开始申请三次，然后在第三个chunk中做出一个0x61的，为做fakechunk做准备：

​ 2.然后连着del两次，紧接着add一次同时覆盖next指针的后两个字节为0x701e，就是tcache中0x100的entry对应的count的位置，因为我们最后造出来的0x100的fakechunk需要放入unsotbin，要求tcahceentry满。

​ 3.然后add一次打出地址，减去0x1e得到heap_base

然后 最关键的一步来了，当时就是这里出问题导致后面del次数不够用，libcbase打出来后就gg了

1.先上一下大师傅的这一步的payload:Add('bb',p64(heap_base + 0x280) + p64(heap_base + 0x268) + p64(0x101) + p64(heap_base + 0x270))

可以看到我们之前已经用0x61给大小为0x100的fakechunk预留了空间，其实这里就是布置fakechunk，0x360-0xf0=0x270.在这个位置放0x101，然后在他的前面放一组指针构成循环链表，如下：

这个fakechunk横跨了0，1，2号三个chunk。

2.接下来看一下构造的那三个指针（在0x270、0x268、0x280），这三个指针构成一个循环链表，变相的增加了我们tcache中可用的chunk

3.接下来我们劫持tcache，同时改0x80的entry里指针指向0x280。效果如下：

可以看到此时0x100的entry已满，同时0x80的entry指针指向heap_base+0x280的位置。效果如下，循环链表形成了：

ciscn_2019_sw_7

题目概览

1.glibc2.27，free后清空表中存储的指针但是不置NULL，不能连续free，考虑劫持tcachebin结构体

2.有一个关键的堆溢出，一开始没看到，unsigned int跟int比较，int自动转为unsigned int，所以只要size传入0，就造成下溢，可以无限读取，堆溢出。

3.限制申请的size在[0x10,0x60]所以需要造fakechunk来拉入unsorbin。

思路

1.初始堆构造如下，free三个，然后拉回来一个size=0的堆溢出，改下面的tcachebin中的 chunk的next指针指向tcache结构体，准备劫持tcache（改next的指针后两个字节指向heapbase+偏移，但是远程需要爆破半个字节）

2.申请到tcache，改0x100的tcache entry的count满（最后我们构造0xe1的fakechunk）

3.再del，add一次size为0的，造成堆溢出，改下方的0x71为0xe1

此时的0xe0的fakechunk由：两个已经一开始free掉的0x60以及一个没有free的0x20构成，此时table中有最后那个没有free的0x20的指针，接下来free这个fakechunk，然后申请两次，下放unsotbin指针到底下那个没有free的0x20chunk里，再调用Show打出来。

4.最后再用一次堆溢出，改mallochook为ogg即可。
