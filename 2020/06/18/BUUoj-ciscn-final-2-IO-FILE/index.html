BUUoj-ciscn_final_2-IO_FILE
Created 2020-06-18| Updated 2020-06-18|Buuoj
Post View:|Comments:

题目概览

  • 标准菜单堆，64位题，没有去掉符号表，保护全开。
  • 首先nop掉alarm
  • 没有edit函数

    Sandbox_Loading

  • 这个函数中调用了prctl函数，然后定义了一大堆变量
  • 函数原型int prctl(int option, unsigned long arg2, unsigned long arg3,unsigned long arg4, unsigned long arg5);
  • 通过阅读手册，这个函数实际上有很多个参数，但是他们都依赖于第一个参数option，在本题中option=38
  • prctl() is called with a first argument describing what to do (with values defined in <linux/prctl.h>), and further arguments with a significance depending on the first one. The first argument can be:
  • 然后第一个参数他给出了很多值，我们重点看一下38:#define PR_SET_NO_NEW_PRIVS 38

option为38，arg2为1时：不能够进行 execve 的系统调用，同时这个选项还会继承给子进程。也就是说我们的system和one_gadget被禁用了，系统调用被限制了
ps: 给出一个师傅绕过prctl沙箱的exp：prctl绕过

init

  • init是一个很重要的函数，在这个函数打开了flag文件描述符，并通过dup2赋给666.

    get_atoi

  • 调用read读入0x10，然后调用atoi转整形返回

    allocate

  • 可以开两种大小的chunk
  • int_pt为malloc返回大chunk的位置（全局变量）
  • short_pt为malloc返回小chunk的位置（全局变量）
  • bool为全局公用的一个标志位。（全局变量）
  • 每次读入inode number会分别存储在chunk中的两个位置

    delete

  • 先判断全局标志位bool值是否被清零
  • 没有被清零，free相应的chunk，但是free后没有置空，清零了全局标志位bool

    show

  • 一个选项是打印两字节，一个是打印四字节
  • 也就是说无法全部打出来，不过也够用。

    bye_bye

  • 这个函数中调用了scanf函数，scanf默认对应的是stdin，其文件描述符为0

  • glibc-2.27

  • 本题基于ubuntu18.04，引入了tcachebin
  • tcachebin优先级高于fastbin、他大幅提高了堆管理的效率，但是舍弃了安全性。。简直是个裸的bin

    tcachebin结构体

    位于：glibc/malloc/malloc.c
    源码分析看这里：m4x师傅

跟fastbin比较有一个很不同的地方：tcachebin中next指针 tcache 的 next 指向 user data 的地方，即 chunk header 之后，而fastbin中fd 指向 chunk 开头的地址

攻击思路（打_IO_FILE）

_IO_FILE概述

  • 可以看到其实show和allocate函数对于大小都有限制，我们只能控制一部分数据。
  • 我们在init函数中打开了flag文件描述符号并赋给了666
  • scanf函数会用到_IO_2_1_stdin_结构体，其实是一个typedef后的_IO_FILE_plus

    程序中打印一下大概就是这样(这里实际上是最后我们已经改掉了_fileno成员后的结果)

  • 我们在最后的bye_bye函数的地方bt一下看看
    可以看到，scanf确实使用了IO_2_1_stdin,并且最底层的时候调用了read
  • 关于_IO_FILE_plus 其实就是对_IO_FILE做了一个封装，其中另外定义了一个虚表指针指向_IO_jump_t 可以动态调试看一下这张表：
    根据我们刚刚bt出来的信息，最终实际上调用了_IO_default_uflow其原型如下： 可以看到传进来的参数是_IO_FILE *类型。
  • 然后我们接着跟踪_IO_UNDERFLOW 最终到达最底层的_IO_new_file_underflow

这个_IO_new_file_underflow函数，是最终调用系统调用的地方，在最终执行系统调用之前，仍然有一些检查，整个流程为： 1. 检查FILE结构体的_flag标志位是否包含_IO_NO_READS，如果存在这个标志位则直接返回EOF，其中_IO_NO_READS标志位的定义是#define _IO_NO_READS 4 / Reading not allowed /。 2. 如果fp->_IO_buf_base位null，则调用_IO_doallocbuf分配输入缓冲区。 3. 接着初始化设置FILE结构体指针，将他们都设置成fp->_IO_buf_base 4. 调用_IO_SYSREAD（vtable中的_IO_file_read函数），该函数最终执行系统调用read，读取文件数据，数据读入到fp->_IO_buf_base中，读入大小为输入缓冲区的大小fp->_IO_buf_end - fp->_IO_buf_base。 5. 设置输入缓冲区已有数据的size，即设置fp->_IO_read_end为fp->_IO_read_end += count。

最终调用了read这个syscall，与我们bt的一致

而此时的FILE其实就是_IO_FILE

好了最终到达最最底层的地方，_IO_SYSREAD即表中的_IO_file_read也是本题的攻击重点，可以看到在调用read的时候传入的就是fp->_fileno！只要我们把这个_fileno劫持为666，scanf就会读出我们想要的flag！

堆操作

这一部分就非常trick了我感觉。就说一下整体的思路 ，细节我就不调试了，因为比较trick

  • 首先申请0x30然后free进tcachebin
  • 然后连开4个0x20，这里并不会把之前那个0x30切割然后拿出来。free后的0x20被放入不同的tcachebin
  • 由于全局变量bool是公用的，当我们delet一次后bool被清0，此时没法继续delet，所以我们要再申请一次，注意申请不同大小的chunk（恢复bool标志位），然后申请完在把另一个大小的chunk进行double free
  • 接下来调用show直接打出此时tcachebin中这个chunk的next指针，然后可以根据chunk的叠放次序计算出首chunk的地址（减去0xa0）
  • 然后我们把这个已经df的chunk用malloc拉回来，然后改他的next指针指向chunk0也就是第一个0x30的块。跟fastbin比较有一个很不同的地方：tcachebin中next指针 tcache 的 next 指向 user data 的地方，即 chunk header 之后，而fastbin中fd 指向 chunk 开头的地址
  • 接下来我们的目标是泄漏地址（libc_base）首先我们需要一个chunk进入unsortedbin才行，这样才能让我们通过主分配区来反算libc基地址
  • 我们现在已经改最后一个0x20的chunk指向第一个0x30的chunk了，接下来我们可以直接编辑这个0x30的chunk，我们在这个chunk的header处做一个fake_size=0x90，这样方便我们后期把它拉入unsortedbin，这也是我们一开始为什么要申请0x30+0x20*3

    更改后：
  • 接下来是我最想吐槽的地方：2.27下的tcachebin就是个裸bin吧。。我们直接把这个0x90的chunk连着free7次，填满tcache后，第八次free把它拉入unsortedbin，没错，就是连着free，这都不能叫double free，这更甚于df
  • 拉入unsortbin后就好说了，直接打出来fd指针减去偏移和libc.sym[‘__malloc_hook’]得到libc_base，这一阶段就完成了
  • 下来打io结构体中的_fileno成员改成666
  • 首先通过libc_base算出_IO_2_1_stdin_运行时的地址
  • 然后切割0x90成0x60（下）0x30（上），改切割出的0x30的指针指向_IO_2_1_stdin_
  • 再切割一次0x30准备做double free
  • 恢复一下bool位，做double free（对于第二次切割出的chunk）完成
  • 可以看到此时首chunk指向_IO_2_1_stdin_我们需要改这个，首先要能写他，这里我们老套路，改df后的chunk的next指针指向首chunk然后连续allocate拿到首chunk，再allocate一次，最终写_IO_2_1_stdin_+0x70也即是_fileno的位置。改为666
  • 最后调用byebye函数，scanf从666中读取flag，下一行打印出来。
